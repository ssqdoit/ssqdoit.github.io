讯飞文档协同编辑能力揭秘 - Delta / OT 🗿 认识协同编辑 🤔 什么是协同编辑 协同编辑文档是指多个用户同时对同一个文档进行编辑操作，不会因为用户并发修改导致冲突、结果不一致甚至数据丢失的问题。

👌 协同编辑好处 • 解决多人编辑可能产生的冲突问题 • 实时自动合并多人编辑的文档 • 提升文档编辑的作业效率

那如何实现协同编辑呢❓ 这里主要用到了协作算法 OT
市场上有很多文档使用了 OT 算法 ，例如： • 讯飞文档 • 石墨文档 • 腾讯文档 • Google Docs

🙋🏻‍♂️ 先来看看有哪些处理冲突的方式

  1. 加锁：某一文档或段落同时只运行一个用户编辑，例如用户 A 在编辑时，就锁住文档，只能 A 进行更新。用户 B 就不能编辑，或编辑后提交修改被服务器丢弃；
  2. 覆盖： 谁最后修改，就全量使用他的修改，更早一些的其他人的修改会被丢弃；
  3. 用户自行处理冲突： 就像 git merge 导致的冲突一样，会提示哪个地方被同时修改了，让合并者手动选择使用哪一个修改；
  4. 使用一致性算法： 比如接下来要介绍的 OT 算法，多用户编辑进行算法处理调整，在多个客户端生成一致的修改结果。

🤖 认识 OT 算法 OT 的英文全称是 Operational Transformation , 是一种基于操作转换的协同冲突处理的算法，OT 协同算法的核心在于操作 Operation 与转换 Transformation ，操作 Operation 指明了所有的操作必须原子化，而转换 Transformation 则指明了所有的操作必须要有转换的方案。 核心思想： 对冲突的操作进行转换，达到最终一致性

举个 🌰 有一个文档大家内容都是 12345 A 用户在第二个字符后面插入了一个 a ，B 用户在第二个字符面插入了一个 b ， 这时候 A 用户这边看到的是 12a345 ， B 用户这边看到的是 12b345 ； 假设 A 用户数据先提交到服务， B 用户数据后到达服务。那预期的数据结构应该为 12ab345 。 在未经过 OT 处理会发生什么 ❓ A： A 在编辑完后，内容是 12a345 , 过了一会 B 编辑了，编辑行为是在第二个字符后面写了一个 b ， 在 A 这边执行这个操作后，最终变成了 12ba345
B： B 在编辑完后，内容是 12b345 , 过了一会 A 编辑了，编辑行为是在第二个字符后面写了一个 a ， 在 B 这边执行这个操作后，最终变成了 12ab345 如下图：

🎯 核心概念 • 基于向量时钟（Vector Clock） 向量时钟用于标记操作的先后顺序和并发性，它能够追踪每个操作的版本和作者，以便在协同编辑过程中解决冲突。 • 基于文本差异（Text Diff） 文本差异算法用于比较两个文本之间的差异，找出需要添加、删除或修改的部分。这对于文本编辑操作的同步和合并非常重要。 • 基于操作序列 / 原子性（Operation Sequence） 操作序列是一系列操作的有序集合，每个操作都包含对文本的修改或变换。OT算法使用操作序列来跟踪和处理并发编辑操作。 • 基于操作转换（Operation Transformation） 操作转换是OT算法的核心概念，它通过将并发操作转换为可并行执行的操作序列来解决冲突。操作转换确保在多个用户同时编辑文本时，各自的操作能够正确应用并保持一致性。

🤏 OT 算法在讯飞文档中的应用 讯飞文档中的文字文档采用的是 Quill 富文本编辑器， 不过我们对 Quill 富文本编辑器做了大量二次开发，例如表格、图片定制、自定义组件、离线、中文支持、移动端适配等等； 富文本编辑器本身也是一个很复杂的模块，但不是本次关注的重点，感兴趣的我们可以会后一起继续探讨 🔬。 如下图 Quill 架构清晰，核心功能简介： • Delta ：富文本数据层，负责描述数据 • Parchment ：富文本控制器层，可以理解为抽象的 dom，负责操控 dom ，将数据变化映射到 UI 表示 • Formats ： Quill 支持多种格式，包括 UI 控件和 API 调用 • Module ：模块化形式，负责解耦功能项，管控模块加载

Quill 富文本编辑器中的 Delta Delta 是 Quill 的数据层，负责表示文本操作和格式化的数据结构。是基于 OT 算法实现，他可以表示插入、删除、转化、格式化等操作。

数据结构 🕸️ Delta 中数据插入类型分为 纯文本 与 embed 嵌入式内容。通过 attributes 区分不同的表现形式，应用场景例如我们常见的 标题、加粗、斜体、列表等。

var content = [{ insert: 'Hello' }, { insert: 'World', attributes: { bold: true } }, { insert: { image: 'https://quilljs.com/logo.png' }, attributes: { width: '100' } }];

如上述代码中 image 就是一个 embed 类型，这种嵌入类型的文本长度为 1 , 类似于一个文字长度。

扁平化数据 Delta 中数据都是扁平的，没有子节点。那他如何表达 文档 Dom 树结构的？ 关键在于 Quill 假定富文本不存在块元素的嵌套，即一行中不能同时存在标题和列表。遇到换行符则新建一行作为块级别 tag open ，直到遇到下一个换行符，作为 tag close 。

紧凑型 必须约定 Delta 中的数据格式标识形式，同一个富文本仅有一种数据表示形式，否则同一种数据可能存在不同表示，例如表示 Hello Word 的不紧凑形式： var content = [ { text: 'Hel' }, { text: 'lo' }, { text: 'World', attributes: { bold: true } } ]; 下面的方式更明确，易于理解和维护： var content = [ { text: 'Hello' }, { text: 'World', attributes: { bold: true } } ];

核心算法 🍗 Push Insert 插入 Retain 保留 Delete 删除 主要是处理富文本操作的一些场景，核心主要是一下： • 连续的 insert 、 retain 、 delete 需要判断做合并操作，对于 insert 、retain 则是 attrs 相同且 insert 都为 string 即可合并，例如 new Delta().insert('hello').insert('world') ，需要合并成 insert: 'hello world' ； • delete 之后的 insert ，会做顺序调整， 规范是放到 insert 后面（因为由于在同一索引处的删除之前或之后插入并不重要，因此始终倾向于先插入）； • 其他场景则直接 push ops 即可。 这主要是为了 ops 数据格式的一致性，为了之后的各种操作运算做逻辑统一化处理。

🍕 Invert 反转操作 返回一个反转的 Delta ，其效果与基础文档 Delta 相反。有以下等式： base.compose(delta).compose(inverted) === base。

🍟 Diff & Compose 对比文档 a 与文档 b 的不同 diff ，有以下等式： a.compose(diff) == b a.diff(b) == diff

compose 合并文档，中有几条原则，决定计算规则： • insert / delete / retain ，类型相同时选择性直接合并； • insert 优先于 delete / retail ； • delete 、 retain 同时存在时，按先后顺序即可，对于 retain -> delete 场景，可直接忽略 retain 操作；

diff 计算文档差异，是基于 fast-diff ( Node.js 纯文本 diff 算法实现，这里不展开)，将 ops 结构转换成纯文本进行对比，只关注 insert string 的部分，其他一律使用占位符替代。更多参考：fast-diff - npm (npmjs.com)。

🍔 Transform 转化 根据已有的 Delta 操作转换给定的 Delta 。 示例如下： const a = new Delta().insert('a'); const b = new Delta().insert('b').retain(5).insert('c');

a.transform(b, true); // new Delta().retain(1).insert('b').retain(5).insert('c'); a.transform(b, false); // new Delta().insert('b').retain(6).insert('c'); 该方法也是解决冲突的核心方法， transform 方法通过比较两个 Delta 对象的操作序列，并根据一定的规则进行变换，从而解决冲突并保持编辑操作的一致性。

📡 拓展：其他协同编辑方案 CRDT（Conflict-free Replicated Data Type，无冲突复制数据类型）算法是一种用于实现分布式系统中的数据同步的算法。它的设计目标是在多个副本之间实现数据的一致性，同时允许并发修改和操作，而无需进行全局的协调和锁定。 CRDT算法的核心思想是通过设计特定的数据结构，使得数据的合并和同步操作可以在不同副本之间进行，而无需中央服务器的参与。CRDT算法主要关注解决并发冲突，确保在多个副本之间的操作不会产生冲突，从而实现最终一致性。 CRDT算法可以分为两类：状态基础（state-based）和操作基础（operation-based）。 • 状态基础：状态基础的CRDT算法通过在每个副本中维护一个状态，并通过合并和比较状态来解决冲突。例如，使用向量时钟（vector clock）来跟踪不同副本之间的操作顺序。 • 操作基础：操作基础的CRDT算法将每个操作应用于所有副本，副本之间通过传递操作来实现状态的同步。例如，使用操作转换（operation transformation）来解决并发操作冲突。 CRDT算法的优点是具有良好的可扩展性和容错性，能够在分布式系统中实现高效的数据同步和协作。它被广泛应用于分布式数据库、实时协同编辑、分布式缓存等领域，为分布式系统提供了一种有效的数据一致性解决方案。
